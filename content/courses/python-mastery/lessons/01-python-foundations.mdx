## Python's Memory Model — How Objects Really Work

Before writing a single line of useful code, you need to understand what Python is actually doing when you type `x = 42`. This mental model will save you hours of debugging confusion later.

In Python, **everything is an object**. An integer is an object. A string is an object. A function is an object. Even `None` is an object. Each object lives somewhere in memory and has three attributes baked into it:

- **Identity** — its unique address in memory (returned by `id()`)
- **Type** — what kind of object it is (returned by `type()`)
- **Value** — the data it holds

When you write `x = 42`, Python does NOT create a variable that *contains* 42. Instead:
1. Python creates an `int` object in memory with value `42`
2. Python binds the *name* `x` to that object

The name is just a label pointing at an object. This distinction is fundamental.

<PythonPlayground code={String.raw`import sys

x = 42
print(f"Value:    {x}")
print(f"Type:     {type(x)}")
print(f"Identity: {id(x)}")          # memory address

# Assign x to y -- both names point to THE SAME object
y = x
print(f"\nAfter y = x:")
print(f"id(x) = {id(x)}")
print(f"id(y) = {id(y)}")
print(f"x is y = {x is y}")          # True -- same object!

# Rebind x to a new value -- y is unaffected
x = 100
print(f"\nAfter x = 100:")
print(f"x = {x}, id(x) = {id(x)}")
print(f"y = {y}, id(y) = {id(y)}")   # y still points to 42
print(f"x is y = {x is y}")          # False now

### Identity vs Equality: `is` vs `==`

This is one of Python's most misunderstood distinctions.

- `==` checks **value equality** — do the objects hold the same data?
- `is` checks **identity** — are they the *exact same object* in memory?

<PythonPlayground code={String.raw`a = [1, 2, 3]
b = [1, 2, 3]   # a new list with same values
c = a            # c points to the SAME list as a

print(f"a == b: {a == b}")    # True  -- same values
print(f"a is b: {a is b}")    # False -- different objects
print(f"a == c: {a == c}")    # True  -- same values
print(f"a is c: {a is c}")    # True  -- same object!

# Mutating through one name affects the other
c.append(4)
print(f"\nAfter c.append(4):")
print(f"a = {a}")   # [1, 2, 3, 4] -- a was affected too!
print(f"c = {c}")   # [1, 2, 3, 4]
`} height={300} />

This is called **aliasing** — two names pointing to the same mutable object. It is a common source of bugs. Always use `==` to compare values. Reserve `is` for checking `None` (`if x is None`).

### Integer Caching — A CPython Optimization

CPython pre-creates integer objects for small integers (-5 to 256) and reuses them. This is why `a is b` is `True` for small numbers but `False` for large ones — it's a performance optimization, not part of the language spec. You should never rely on this behavior.

<PythonPlayground code={String.raw`# Small integers are cached -- same object
a = 100
b = 100
print(f"a is b (100): {a is b}")    # True -- cached

# Large integers are NOT cached
x = 1000
y = 1000
print(f"x is y (1000): {x is y}")  # False -- different objects

# But equality still works
print(f"x == y: {x == y}")          # True

# This is why you should NEVER use 'is' to compare numbers
# Always use == for value comparison
`} height={260} />

### Reference Counting and Garbage Collection

Python tracks how many names (references) point to each object. When the count hits zero, the object is freed. You can inspect this with `sys.getrefcount()` — it reports one higher than you expect because the function call itself creates a temporary reference.

<PythonPlayground code={String.raw`import sys

x = [1, 2, 3]
print(f"refcount of x: {sys.getrefcount(x)}")  # 2 (x + getrefcount arg)

y = x   # add a reference
print(f"refcount after y=x: {sys.getrefcount(x)}")  # 3

z = x   # add another
print(f"refcount after z=x: {sys.getrefcount(x)}")  # 4

del y   # remove a reference
print(f"refcount after del y: {sys.getrefcount(x)}")  # 3

# When refcount hits 0, the memory is reclaimed
# CPython also has a cyclic garbage collector for reference cycles
# You rarely need to think about any of this -- Python handles it
`} height={280} />

**Short-circuit evaluation** is essential to understand — Python stops evaluating `and`/`or` the moment the result is known:

<PythonPlayground code={String.raw`# 'and' returns first falsy value, or the last value

print(None or "fallback") # "fallback"
print(0 or 0 or 0)        # 0   (all falsy, return last)

## Strings — A Complete Deep Dive

### Immutability and Why It Matters

Strings in Python are **immutable** — once created, they cannot be modified. Any "modification" operation creates and returns a new string object. This enables safe sharing (two names can point to the same string with no aliasing issues) and allows strings to be hashable (usable as dictionary keys).

<PythonPlayground code={String.raw`s = "hello"

# += does NOT modify the string in-place -- it creates a new string
s = s + " world"
print(f"id after:  {id(s)}")
print(f"s = {s}")

# You CANNOT change individual characters
# s[0] = "H"  # TypeError: 'str' object does not support item assignment

# To build strings from many parts, join() is the right tool
# It creates exactly one string object
parts = ["The", "quick", "brown", "fox"]
result = " ".join(parts)
print(f"\nJoined: {result}")

# Loop concatenation is inefficient -- creates N intermediate strings
slow = ""
for word in parts:
    slow += word + " "           # creates a new object each iteration

# join() is O(n) and creates one object
fast = " ".join(parts)
print(f"Fast result: {fast}")

def letter_grade(avg):
    if avg >= 90: return "A"
    elif avg >= 80: return "B"
    elif avg >= 70: return "C"
    elif avg >= 60: return "D"
    else: return "F"

def std_dev(scores):
    mean = sum(scores) / len(scores)
    variance = sum((s - mean) ** 2 for s in scores) / len(scores)
    return math.sqrt(variance)

    filled = int(avg / 100 * width)
