## Part 1: Python Exception System Internals

Every Python program you write will eventually fail. The question is not *if* an error occurs but *how gracefully you handle it*. Python's exception system is one of the most expressive in any language — deeply object-oriented, fully inspectable, and designed to be extended. Understanding it at the internals level transforms error handling from defensive boilerplate into a deliberate design tool.

### The Exception Hierarchy Tree

Python's exceptions form a strict inheritance hierarchy rooted at `BaseException`. Every exception class you can catch or raise is a node in this tree:

```
BaseException
├── SystemExit                 ← raised by sys.exit()
├── KeyboardInterrupt          ← Ctrl+C
├── GeneratorExit              ← generator.close() / garbage collection
└── Exception                  ← all "normal" exceptions
    ├── ArithmeticError
    │   ├── ZeroDivisionError
    │   ├── OverflowError
    │   └── FloatingPointError
    ├── LookupError
    │   ├── IndexError
    │   └── KeyError
    ├── ValueError
    ├── TypeError
    ├── AttributeError
    ├── NameError
    │   └── UnboundLocalError
    ├── ImportError
    │   └── ModuleNotFoundError
    ├── OSError  (also IOError, EnvironmentError — aliases)
    │   ├── FileNotFoundError
    │   ├── PermissionError
    │   ├── IsADirectoryError
    │   ├── FileExistsError
    │   ├── TimeoutError
    │   └── ConnectionError
    │       ├── ConnectionResetError
    │       └── ConnectionRefusedError
    ├── RuntimeError
    │   └── RecursionError
    ├── StopIteration
    ├── StopAsyncIteration
    ├── MemoryError
    ├── BufferError
    ├── EOFError
    ├── SyntaxError
    │   └── IndentationError
    │       └── TabError
    ├── UnicodeError
    │   ├── UnicodeDecodeError
    │   ├── UnicodeEncodeError
    │   └── UnicodeTranslateError
    ├── Warning
    │   ├── DeprecationWarning
    │   ├── UserWarning
    │   └── RuntimeWarning
    └── ExceptionGroup (Python 3.11+)
```

The critical insight: `except Exception` catches everything *except* `SystemExit`, `KeyboardInterrupt`, and `GeneratorExit`. Those three inherit directly from `BaseException` because they represent program lifecycle events, not errors in your logic. Accidentally catching `KeyboardInterrupt` would make your program unresponsive to Ctrl+C.

### How Exceptions Propagate: Call Stack Unwinding

When an exception is raised, Python does *call stack unwinding*: it travels up the call stack frame by frame, looking for a matching `except` handler. If none is found at the outermost frame, the program prints a traceback and terminates.

<PythonPlayground code={String.raw`import traceback

    # This is where the exception originates
    result = 10 / 0

def level_2():
    level_3()

def level_1():
    level_2()

# Capture the traceback without terminating
try:
    level_1()
except ZeroDivisionError as e:
    print("Caught at the top level!")
    print(f"Exception type: {type(e).__name__}")
    print(f"Exception args: {e.args}")
    print(f"Exception str:  {str(e)}")
    print()
    print("Full traceback:")
    traceback.print_exc()

The traceback shows the full path — `level_1` called `level_2` called `level_3`, which is where the exception originated. Each frame has its own local variables (accessible via `traceback.extract_tb`).

### try / except / else / finally — Every Clause

Each block has a specific semantic purpose. Using them correctly communicates intent to readers.

<PythonPlayground code={String.raw`import io

def process_data(raw):
    """Demonstrate all four clauses of try/except/else/finally."""
    file_opened = False
    result = None

    try:
        # The try block: code that might raise
        print(f"[try]     Attempting to parse: {repr(raw)}")
        value = int(raw)
        if value < 0:
            raise ValueError(f"Value must be non-negative, got {value}")
        result = value * 2

    except ValueError as e:
        # except: runs ONLY if an exception was raised in try
        print(f"[except]  Caught ValueError: {e}")
        result = 0

    else:
        # else: runs ONLY if NO exception was raised
        # Perfect for code that should only run on success
        print(f"[else]    No exception — result is {result}")

    finally:
        # finally: ALWAYS runs, even with return/break/continue/raise
        # Use for cleanup: close files, release locks, end transactions
        print(f"[finally] Cleanup — result so far: {result}")

    return result

print("=== Case 1: valid positive input ===")
r = process_data("42")
print(f"Returned: {r}")
print()

print("=== Case 2: invalid string ===")
r = process_data("hello")
print(f"Returned: {r}")
print()

print("=== Case 3: negative number ===")
r = process_data("-5")
print(f"Returned: {r}")
`} height={500} />

**The `else` clause is underused.** It was designed exactly for "code that should only run when the `try` block succeeded." Using `else` instead of putting success code inside `try` avoids accidentally catching exceptions raised by the success code itself.

**The `finally` clause always runs.** Even if you `return` inside `try` or `except`, Python executes `finally` before actually returning. This makes it the right place for resource cleanup.

## Part 3: Custom Exceptions — Professional Patterns

### Why Custom Exceptions

Using Python's built-in exceptions for domain errors is a code smell. When a library raises `ValueError`, callers cannot distinguish between "bad argument to int()" and "business rule violation." Custom exceptions create a precise, testable contract.

<PythonPlayground code={String.raw`# Basic custom exception with extra attributes
class AppError(Exception):
    """Base for all application errors. Adds error_code."""
    def __init__(self, message, error_code=None):
        super().__init__(message)
        self.error_code = error_code
        self.message = message

    def __str__(self):
        if self.error_code:
            return f"[{self.error_code}] {self.message}"
        return self.message

    def __repr__(self):
        return f"{type(self).__name__}({self.message!r}, error_code={self.error_code!r})"


class ValidationError(AppError):
    """Input failed validation. Adds field and invalid_value."""
    def __init__(self, field, invalid_value, reason):
        message = f"Field '{field}' is invalid: {reason}"
        super().__init__(message, error_code="VALIDATION_ERROR")
        self.field = field
        self.invalid_value = invalid_value
        self.reason = reason


class DatabaseError(AppError):
    """Database operation failed."""
    pass

class RecordNotFoundError(DatabaseError):
    """A specific record was not found."""
    def __init__(self, model, record_id):
        super().__init__(
            f"{model} with id={record_id} not found",
            error_code="NOT_FOUND"
        )
        self.model = model
        self.record_id = record_id

class DuplicateRecordError(DatabaseError):
    """Attempted to create a duplicate record."""
    pass


# Use the hierarchy
def create_user(name, age):
    if not name or not isinstance(name, str):
        raise ValidationError("name", name, "must be a non-empty string")
    if not isinstance(age, int) or age < 0 or age > 150:
        raise ValidationError("age", age, "must be integer 0-150")
    # Simulate duplicate
    if name == "admin":
        raise DuplicateRecordError(
            f"User with name='admin' already exists",
            error_code="DUPLICATE"
        )
    return {"id": 1, "name": name, "age": age}


# Callers can catch at any granularity
for args in [("Alice", 30), ("", 25), ("Bob", -5), ("admin", 20)]:
    try:
        user = create_user(*args)
        print(f"Created: {user}")
    except ValidationError as e:
        print(f"Validation failed on '{e.field}': {e.reason}")
    except DuplicateRecordError as e:
        print(f"Duplicate: {e}")
    except DatabaseError as e:
        print(f"DB error: {e}")
    except AppError as e:
        print(f"App error [{e.error_code}]: {e}")

## Part 4: Context Managers — Complete Mastery

### The `with` Statement Protocol

A context manager is any object with `__enter__` and `__exit__` methods. The `with` statement calls `__enter__` on entry and `__exit__` on exit — even if an exception is raised.

<PythonPlayground code={String.raw`# Build a context manager from scratch
import time
import traceback

class Timer:
    """Context manager that measures elapsed time."""

    def __init__(self, name="block"):
        self.name = name
        self.elapsed = None

    def __enter__(self):
        # Called on entry to the with block
        # Return value is bound to the 'as' variable
        self._start = time.perf_counter()
        print(f"[Timer] Starting '{self.name}'")
        return self  # bound to 'as timer'

    def __exit__(self, exc_type, exc_val, exc_tb):
        # Called on exit — always
        # exc_type, exc_val, exc_tb are None if no exception
        self.elapsed = time.perf_counter() - self._start
        print(f"[Timer] '{self.name}' finished in {self.elapsed*1000:.3f}ms")

        if exc_type is not None:
            print(f"[Timer] Exception occurred: {exc_type.__name__}: {exc_val}")
            # Return True to SUPPRESS the exception
            # Return False/None to PROPAGATE it
            return False  # propagate

        return False

# Normal usage
with Timer("sum computation") as t:
    total = sum(range(1_000_000))
print(f"Sum = {total}, elapsed = {t.elapsed*1000:.3f}ms")
print()

# Exception inside with block
try:
    with Timer("failing operation"):
        x = 1 / 0
except ZeroDivisionError:
    print("ZeroDivisionError propagated (as expected)")

## Part 5: Logging — Production Patterns

### Why logging beats print()

`print()` is for exploratory debugging. `logging` is for production. Here is why:

- **Levels**: DEBUG/INFO/WARNING/ERROR/CRITICAL — filter verbosity without code changes
- **Destinations**: Console, file, network, syslog — without changing call sites
- **Format**: Timestamps, module names, line numbers — free with formatters
- **Off switch**: Set level to WARNING in production — all DEBUG/INFO disappear with no performance hit
- **Thread-safe**: The logging module is thread-safe; print() is not guaranteed

def process_csv(csv_content, strict=False):
    """
    Process CSV content string.
    strict=True: raise on first error
    strict=False: collect errors, return (records, errors)
    """
    with processing_context("CSV processing") as error_log:
        reader = csv.DictReader(io.StringIO(csv_content))

        # Validate header
        if not reader.fieldnames:
            raise FileFormatError("input", "CSV", "empty file")

        fields = set(reader.fieldnames)
        missing = REQUIRED_COLUMNS - fields
        if missing:
            raise FileFormatError("input", "CSV",
                f"missing columns: {', '.join(sorted(missing))}")

        records = []
        for row_num, row in enumerate(reader, start=2):
            try:
                clean = validate_record(row_num, row)
                records.append(clean)
                logger.debug(f"Row {row_num}: valid — {clean['name']}")

                logger.warning(f"Row {row_num}: validation error — {e}")
                error_log.append(e)
                if strict:
                    raise

        return records, list(error_log)
